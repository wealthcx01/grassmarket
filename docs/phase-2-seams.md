# Phase-2 / Phase-3 integration seams

Grassmarket is built standalone today but every cross-system boundary was designed as a **seam**, not
a wall. This document is the map of those seams: what the phase-2 (Holy Corner) and phase-3
(Viewforth) projects consume from Grassmarket, and where the code changes when they arrive. Nothing
here is speculative architecture — each seam already exists in the codebase and is load-bearing now.

Normative source: PRD §1 (portals/domains), §8 (tech), CLAUDE.md #4 (contract-typed), #5 (repository
layer), #7 (two-track / value bridge), #8 (AI-proposes gate).

## The three portals (PRD §1)

| Portal | Domain | Project | Phase |
|---|---|---|---|
| **Advisor Studio** (this repo) | `advisors.bruntsfieldcapital.com` | Grassmarket | 1 — now |
| **Bruntsfield OS** | `os.bruntsfieldcapital.com` | Holy Corner (Elite Vault System adaptation) | 2 |
| **Client portal** | `clients.bruntsfieldcapital.com` | Viewforth | 3 |

The main site routes login by role. Grassmarket issues JWTs today with a claim shape that already
matches the future Holy Corner SSO (see below), so the auth seam is the cheapest to cross.

## Seam 1 — `bcap-contracts` is the shared surface

`packages/bcap_contracts` is described in its own docstring as "the future Holy Corner API surface."
It is the single source of truth for every resource that crosses a system boundary:

- **Pydantic v2 models** → **JSON Schema** (`packages/.../json_schema/*.json`, regenerated by
  `scripts/generate_schemas.py`) → **generated TypeScript types** for the frontends.
- **Schemas win on conflict** (CLAUDE.md #4). Both Grassmarket's Next.js app and any phase-2/3
  frontend consume the same generated types, so a contract change is a single edit that ripples
  everywhere with a schema-parity check in CI.
- The registry (ADR-0001: one scale, one key registry, unknown key = refusal) and ADR-0002 (score
  and currency never mix) are structural invariants of the package, so any consumer inherits them.

**What Holy Corner consumes:** the resource models it will serve from its own API — `Consultant`,
`Invitation`, `Prospect`/pipeline, `EngagementStatus`, `ScoringRun`, `CoefficientSet`, `AuditEvent`,
earnings/commission, workbench (certification/arena/bench). These are already the shapes Grassmarket
persists and returns.

**What Viewforth reads:** the **client-facing** artifacts only — client-usable deliverables (gated
by GRS-0015 / `client_usable=True`), the three-layer value bridge output (cost £ / lever NPV £ /
strategic ordinal, ADR-0002), and de-identified immutable scoring runs. Viewforth never sees a draft
pack: the `resolve_mode` + `assert_uncertainty_client_usable` gate (GRS-0033) refuses to emit a
client artifact from a non-client-usable coefficient set OR uncertainty model.

## Seam 2 — the repository layer swaps its backing store

`src/grassmarket/data/repository.py` is the **only** data-access layer (CLAUDE.md #5). Feature code
never issues a query directly; it calls the repository, whose interface "is shaped like Holy Corner's
future API resources so the backing store can swap from local Postgres to the Holy Corner API without
touching feature code."

The phase-2 change is therefore localised: implement a repository backed by the Holy Corner API (or
put Holy Corner in front of the same Postgres) and inject it. The three properties that make this
safe are already enforced and tested here:

- **Absolute data scoping** (CLAUDE.md #9): `_assert_can_access` / `_require_self_or_admin` — a
  consultant sees only their own pipeline, engagements, earnings, assessments. Holy Corner inherits
  this contract; the scoping tests move with the interface.
- **Fail-loud** (CLAUDE.md #3): no `.get(key, default)` in the scoring path; a missing/unknown key
  refuses rather than defaults.
- **Immutable, versioned scoring runs** (CLAUDE.md #6): append-only, content-hashed, stored with
  engine/methodology/coefficient versions. A run means the same thing on either side of the seam.

## Seam 3 — the JWT claim shape is the SSO contract

`bcap_contracts.auth.JWTClaims` is already the "Holy Corner claim shape" (PRD §1, `.env.example`
comment). Fields: `sub` (consultant id — the scoping principal), `email`, `role`, `tier`,
`assessor_level`, `iss` (`advisors.bruntsfieldcapital.com`), `aud` (`bruntsfield`), `iat`, `exp`.

Phase-2 SSO does not change the claim shape — Holy Corner becomes the issuer (`iss`) and Grassmarket
verifies its tokens. Because the repository scopes on `sub` and the role/tier/assessor-level
vocabularies live in `bcap-contracts`, no feature code changes: only the token *verification* source
(the `iss`/signing key) moves.

## Seam 4 — the AI-proposes / human-approves boundary (unchanged across phases)

Every AI output (meeting-intelligence extraction, deliverable first drafts, practice-arena feedback)
is gated by an approval boundary (CLAUDE.md #8) enforced at runtime, not by convention:
`assert_narratives_approved` refuses a client pack carrying an unapproved AI narrative;
extraction/deliverable drafts are proposals until a consultant signs off. Holy Corner and Viewforth
inherit this guarantee through the same gated artifacts — a client portal can only ever be handed
material a human has approved.

## What is NOT a seam (deliberately deferred)

- **Holy Corner integration itself** — "deferred by design; the contracts package keeps the seam
  clean. Elite Vault System adaptation is the phase-2 project" (PRD §, deferred-scope table).
- **Viewforth client portal** — phase 3; reads only the gated client artifacts above.

The test of a good seam is that crossing it is an additive change on the far side, not a rewrite on
this one. For each seam above, the Grassmarket-side code that changes at integration time is: Seam 1
— nothing (consumers import the package); Seam 2 — one repository implementation, injected; Seam 3 —
the token-verification issuer/key; Seam 4 — nothing. That is the whole point.
